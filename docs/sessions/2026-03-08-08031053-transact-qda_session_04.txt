  Das spezifische Problem eures Datenmodells

  Das entscheidende Constraint ist: Relations sind keine Kanten, sondern First-Class-Objekte. In Standard-Graph-Bibliotheken sind Edges visuell
  dünn und passiv. Hier haben Relations eigene Inscriptions, Designations, Provenance, und können selbst Endpunkte von Meta-Relations sein
  (Eta-Beziehung).

  Das bedeutet: Relations müssen visuell als Nodes auf einer Kante dargestellt werden — mit eigenem Interaktionsbereich, eigenem
  Designation-Dot, eigenem Stack. Der Diamond-Marker aus dem Plan.

  Meine Empfehlung

  ELK.js + Custom Svelte SVG (mit d3-zoom für Pan/Zoom).

  Begründung:
  - ELK.js ist die einzige Engine, die eure Graph-Topologie sauber berechnen kann — Compound Nodes für Phases, Ports für gerichtete Relations,
  Hyperedges für Meta-Relations
  - Custom SVG ist nötig, weil kein off-the-shelf Renderer die visuellen Anforderungen abbildet: Designation-Gradients, Provenance-Icons,
  Meta-Relation-Diamonds, perspectival Collapse-Indikatoren, Phase-Regions
  - d3-zoom für Pan/Zoom ist ~30 Zeilen Code und battle-tested

  SvelteFlow wäre die schnellere Alternative (Canvas-Infrastruktur geschenkt), aber ihr würdet gegen seine Edge-Semantik kämpfen, sobald
  Relations visuell gleichberechtigt werden sollen.

  Die Bibliothek-Kombination wäre also:
  elkjs         → Layout-Berechnung
  d3-zoom       → Pan/Zoom/Viewport
  d3-shape      → Bezier-Kurven für Relations
  Svelte SVG    → Rendering (each Node/Relation ist eine Svelte-Komponente)

  Soll ich einen Prototyp in diese Richtung bauen, oder möchtest du erst über die visuellen Paradigmen der vier Map-Typen (situational,
  social-worlds, positional, network) diskutieren?

 Phases as Visual Cluster Regions on the Canvas                                                                                               

 Context

 Phases currently appear only in a sidebar list. On the canvas, elements assigned to phases have no visual grouping. In Clarke's Situational
 Analysis, ordered maps ARE clusterings — phases should structure the visual space as recognizable, labeled regions.

 The key challenge: a naming can belong to multiple phases with different collapse indices. ELK compound nodes enforce strict hierarchy (one
 parent). So we use a post-layout approach: compute region bounds from member positions after layout.

 Design Decisions

 1. Data: Add phase_ids text[] array to each appearance via a correlated subquery in getMapAppearances. No new API endpoints needed.
 2. Layout: No ELK changes. Lay out flat, compute region bounds post-layout as a pure function of positions + membership.
 3. Regions: Bounding ellipse of member positions + padding. Uses existing CanvasRegion.svelte.
 4. Phase positions: Always derived from member positions, not stored.
 5. Multi-membership: Overlapping translucent ellipses — overlapping areas become slightly more saturated, providing a natural visual cue.
