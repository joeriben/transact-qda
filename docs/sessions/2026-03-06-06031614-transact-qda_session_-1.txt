### 1. "Nativ serverless" vs. Miro-Modell

**Serverless** heißt: Ihr Code läuft als Funktionen auf Cloudflare's Edge-Netzwerk. Kein eigener Server, keine Wartung, automatische Skalierung, Pay-per-Request. **Aber**: JavaScript/TypeScript only, Ausführungszeitlimits (30s–), begrenzter Speicher.

**Das Miro-Modell** (cloud-native, real-time kollaborativ) braucht:
- WebSocket-Verbindungen für Echtzeit-Zusammenarbeit
- Persistente Server-Logik
- Komplexe Graph-Queries

Das spricht **gegen** reines Serverless und **für** einen richtigen Server. Die ehrliche Antwort: Für eine Miro-artige QDA-Anwendung brauchen Sie einen Application Server (Python oder Node.js), z.B. auf Fly.io, Railway oder einem VPS. Cloudflare davor als CDN/Reverse Proxy/DDoS-Schutz – ja. Aber die Anwendungslogik läuft nicht sinnvoll in Workers allein.

Damit relativiert sich der Vorteil von TypeScript-Full-Stack erheblich. **Beide Stacks sind gleich gut möglich**, wenn ohnehin ein eigener Server läuft.

### 2. Datenbank – ehrliche Analyse

Sie haben Recht, Neo4j gehört hier rein. Hier die sachliche Analyse:

| | **Neo4j** | **PostgreSQL + Apache AGE** | **SurrealDB** |
|---|---|---|---|
| **Graph-Modell** | Nativ, zweckgebaut. Cypher-Query-Language ist für Graphen gemacht. | AGE-Extension fügt Cypher auf PostgreSQL hinzu. Graph-Queries möglich, aber aufgesetzt. | Nativ multi-model (Graph + Document + Relational). `RELATE`-Syntax ist elegant. |
| **Flache Ontologie** | Perfekt: Alles ist Node, Relationen sind Edges die selbst Properties haben. Edges können aber nicht selbst Endpunkte anderer Edges sein (→ Hyperedge-Workaround nötig). | Mit AGE möglich, aber zwei Welten (SQL + Cypher) nebeneinander. | Record-Links sind first-class. Flexibelstes Modell für die Anforderung. |
| **Reife/Stabilität** | Industriestandard seit 2007. Riesige Community. | PostgreSQL: absolut. AGE: Apache-Inkubator, weniger battle-tested. | Jung (v1.x seit 2023). Kleinere Community, Stabilitätsrisiko. |
| **Hosting** | Neo4j Aura (managed cloud, free tier vorhanden) oder self-hosted (Java, ressourcenhungrig). | Überall: Neon, Supabase, jeder VPS. | Self-hosted oder SurrealDB Cloud. |
| **Für Miro-artige Collab** | Gut, aber kein eingebauter Realtime-Layer. | Gut, gute Tooling-Integration (Supabase Realtime etc.) | Hat built-in Live-Queries. |
| **Risiko** | Licensing: Community Edition = GPL. Enterprise = kommerziell. | Niedrig (PG) + mittel (AGE). | Höher wegen Jugend des Projekts. |

**Meine ehrliche Empfehlung**: **Neo4j** ist konzeptionell am saubersten für Ihr Datenmodell. Allerdings: Wenn Relationen wirklich *identisch* zu Entitäten sein sollen (d.h. eine Relation kann selbst Ausgangspunkt einer Relation sein), braucht man ein **Hypergraph-Modell** – und da stößt auch Neo4j an Grenzen. In Neo4j wäre der Workaround: Relationen als Nodes modellieren, die über Meta-Edges verbunden werden. Das ist aber ein Pattern, kein Hack.

**Alternative Perspektive**: Vielleicht ist die technisch sauberste Lösung ein **eigenes Graph-Modell auf PostgreSQL** – eine Tabelle `elements` (sowohl Entitäten als auch Relationen) und eine Tabelle `connections` (verbindet beliebige Elemente). Einfach, transparent, kein Vendor-Lock-in, und Ihr ontologisches Prinzip wird *direkt* in der Datenstruktur abgebildet.

### Das Problem mit konventionellen Datenmodellen

In jeder normalen Datenbank gilt: Erst Entität anlegen, dann Relationen zuweisen. Die Entität existiert unabhängig – das ist genau die **self-actional** bzw. **inter-actional** Ontologie, die Dewey/Bentley kritisieren. Die Entität ist vor der Relation da, hat intrinsische Eigenschaften, und wird nachträglich "verbunden".

### Was Dewey/Bentley (und Sünkel) verlangen

Im **trans-actional** Modus gilt: Die Relata werden *durch* die Transaktion konstituiert. Ein "Naming" ist kein Etikett auf ein vorgegebenes Ding, sondern ein funktionaler Akt innerhalb eines Forschungsprozesses, der das Benannte als *dieses bestimmte* überhaupt erst hervorbringt. Dasselbe Phänomen kann in unterschiedlichen relationalen Kontexten unterschiedlich benannt sein – und das sind dann unterschiedliche Entitäten.

### Informatisch abbildbar? Ja.

Der Schlüssel: **Der Grundbaustein ist nicht die Entität, sondern der Benennungsakt (Naming Event).**

**Modell:**

```
events (
  id          UUID PRIMARY KEY,
  type        TEXT,        -- 'naming', 'relating', 'coding', 'mapping' ...
  created_at  TIMESTAMP,
  created_by  UUID,        -- Forschende/r
  context     UUID,        -- Projekt, Map, Coding-Session ...
  data        JSONB
)

elements (
  id              UUID PRIMARY KEY,
  label           TEXT,
  constituted_by  UUID REFERENCES events(id),  -- der Benennungsakt
  -- KEINE intrinsischen Eigenschaften außerhalb
  -- relationaler Kontexte
)

-- Eigenschaften sind relational kontextuiert:
element_aspects (
  id          UUID PRIMARY KEY,
  element     UUID REFERENCES elements(id),
  context     UUID REFERENCES elements(id),  -- in welcher Relation/Map/Situation
  properties  JSONB                           -- Eigenschaften IN DIESER Hinsicht
)
```

**Was das leistet:**

1. **Jedes Element ist auf seinen konstitutiven Benennungsakt rückführbar** – es gibt keinen "nackten" Knoten ohne Entstehungskontext
2. **Eigenschaften sind kontextgebunden** (element_aspects): Dasselbe Element hat verschiedene Bestimmungen je nach relationaler Einbettung – es IST etwas anderes auf der Situational Map als auf der Positional Map
3. **"Entfernt man alle Relationen, verschwindet die Entität"** – nicht physisch gelöscht, aber eine Entität ohne Aspekte/Relationen hat *keinen Inhalt*. Sie ist leere Form.
4. **Naming ist selbst ein Event**, der im Forschungsprozess verortet ist (wer hat wann in welchem Kontext benannt?) – das macht den Forschungsprozess selbst reflexiv nachvollziehbar

### Konsequenz für die UX

Wenn jemand ein Element auf einer Situational Map anlegt, passiert informatisch:
1. Ein `naming event` wird erzeugt (Kontext: diese Map, dieser User, jetzt)
2. Ein `element` wird erzeugt, `constituted_by` → dieses Event
3. Ein `element_aspect` wird erzeugt: die Eigenschaften dieses Elements *in dieser Map*

Wird dasselbe Element auf einer anderen Map referenziert, entsteht ein neuer Aspekt – keine Kopie, aber eine neue kontextgebundene Bestimmung.
