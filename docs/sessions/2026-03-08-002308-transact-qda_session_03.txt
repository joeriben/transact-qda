● Codes sind bereits Namings — mit code-system als Perspektive, Hierarchie über
  Participations (contains), und Annotations als gerichtete Relationen (valence:

  Die Frage "wie damit umgehen" hat m.E. drei Ebenen:

  1. Ontologisch: Codes = Phases = Namings

  Im Datenmodell gibt es keinen strukturellen Unterschied. Ein Code ist ein
  Naming, das aus der code-system-Perspektive als Entity erscheint. Eine Phase
  ist ein Naming, das als Sub-Perspektive einer Map erscheint. Beides sind
  perspektivische Ordnungen.

  Der Unterschied ist die Herkunft: Codes kommen aus der Textarbeit (Dokument →
  Annotation → Code), Phases aus der Map-Arbeit (Situation → Gruppierung →
  Phase).

  2. Praktisch: Brücke fehlt

  Aktuell leben Codes und Maps in getrennten Welten:
  - Codes annotieren Dokumente, haben eine Baumhierarchie
  - Map-Elemente stehen auf Maps, haben Designations und Collapses
  - Es gibt keine Verbindung: ein Code kann nicht als Element auf einer Map
  erscheinen, und ein Map-Element kann nicht als Code auf ein Dokument angewandt
   werden

  3. Was tun?

  Option A: Codes auf Maps bringen. Ein Code (z.B. "Digitalisierungsdiskurs")
  könnte als Element auf einer Situational Map erscheinen — mit eigenem
  Designation-Stack, relationierbar mit anderen Elementen. Technisch: eine
  Appearance des Code-Namings auf der Map-Perspektive.

  Option B: Map-Elemente als Codes nutzbar machen. Ein Map-Element (z.B.
  "Schulbuch") könnte als Code auf Textpassagen angewandt werden. Technisch:
  eine Annotation, die das Map-Element als code_id referenziert.

  Option C: Beides. Codes und Map-Elemente sind dasselbe — man kann jedes Naming
   auf einer Map platzieren UND als Code verwenden. Die Perspektive (Code-System
   vs. Map) bestimmt, wie es erscheint.

  Option C ist ontologisch konsequent, aber — wie du bei Phases gesagt hast —
  die Komplexität steigt. Was ist dein Instinkt?

  Das Problem ist real

  Die Unterscheidung "was kommt aus dem Datenkorpus, was nicht" ist
  methodologisch zentral — nicht ontologisch, aber für die Nachvollziehbarkeit.
  Clarke verlangt, dass die Situational Map ALLES aufnimmt (auch
  Außer-Empirisches), aber die Frage der Herkunft bleibt: Woher stammt dieses
  Naming? Wer muss das wissen? Gutachter, Betreuer, man selbst in sechs Monaten.

  Gegen "Non-Documents" als Lösung

  Die Idee, nicht-empirische Namings über "Non-Documents" zu begründen, erzeugt
  eine Begründungsbürokratie, die dem eigentlichen Problem ausweicht. Man würde
  Quasi-Dokumente erfinden, um eine Lücke zu füllen, die eigentlich eine Lücke
  der Provenienz-Sichtbarkeit ist.

  Was ich stattdessen vorschlagen würde

  Die Unterscheidung ist nicht ontologisch (Code vs. Naming), sondern
  relational: Hat dieses Naming eine Verankerung in Dokumenten oder nicht? Das
  ist bereits im Datenmodell sichtbar — ein Naming mit Annotations (Appearance
  valence: 'codes') ist empirisch verankert, eines ohne ist es nicht.

  Drei Herkunftstypen, alle legitim in Situational Analysis:

  ┌─────────────┬──────────────────────┬─────────────────────────────────────┐
  │  Herkunft   │    Sichtbar durch    │              Beispiel               │
  ├─────────────┼──────────────────────┼─────────────────────────────────────┤

  │             │ Naming)              │ Mapping-Prozess, Memo erklärt warum │
  ├─────────────┼──────────────────────┼─────────────────────────────────────┤
  │             │                      │ "Implizierter Akteur" aus           │
  │ Theoretisch │ Memo + Designation   │ Clarke-Framework, bewusst           │
  │             │                      │ eingebracht                         │
  └─────────────┴──────────────────────┴─────────────────────────────────────┘

  Provenienz-Kette: Welche Participations hat dieses Naming? Führt eine davon zu
