│ Plan to implement                                                            │
│                                                                              │
│ transact-qda: QDA-Software for Situational Analysis                          │
│                                                                              │
│ Context                                                                      │
│                                                                              │
│ Build a QDA (Qualitative Data Analysis) software from scratch,               │
│ methodologically grounded in Situational Analysis (Adele Clarke) and         │
│ ontologically grounded in transactional philosophy (Dewey/Bentley) and       │
│ Sünkel's relational educational theory (Eta-Beziehung). The software is      │
│ tailored to research at the intersection of STS and Erziehungswissenschaft.  │
│                                                                              │
│ The core ontological commitment: entities are constituted through            │
│ relational/naming acts, not pre-existing. Relations are first-class objects  │
│ that can themselves be related to other elements (zeugmatic relations).      │
│ Properties are context-bound, not intrinsic.                                 │
│                                                                              │
│ Tech Stack                                                                   │
│                                                                              │
│ ┌────────────┬──────────────────────┬─────────────────────────────────────┐  │

│ │ Database   │ JSONB                │ GIN-indexed JSONB for flexible      │  │

│                                                                              │
│ Data Model (Transactional Ontology)                                          │
│                                                                              │
│ Core Principle                                                               │
│                                                                              │
│ The fundamental unit is the event (naming/relating act), not the entity.     │
│ Elements are constituted by events and have context-bound properties         │
│ (aspects).                                                                   │
│                                                                              │
│ Schema Overview                                                              │
│                                                                              │
│ -- Every mutation is an event (the naming/relating act)                      │
│ events (                                                                     │
│   id UUID PK, project_id, type TEXT, created_by, created_at,                 │
│   context_id UUID, data JSONB, seq BIGSERIAL                                 │
│ )                                                                            │
│ -- type: 'element.name', 'element.relate', 'element.update',                 │
│ 'element.delete',                                                            │
│ --       'aspect.set', 'code.create', 'code.apply', 'memo.create',           │
│ 'map.create',                                                                │
│ --       'ai.suggest', 'ai.accept', 'ai.reject', ...                         │
│                                                                              │
│ -- Everything is an element: entities, relations, codes, memos, maps,        │
│ documents                                                                    │
│ elements (                                                                   │
│   id UUID PK, project_id, kind TEXT, label TEXT,                             │
│   constituted_by UUID -> events(id),  -- the naming act that created this    │
│   source_id UUID -> elements(id),     -- NULL for non-relations              │
│   target_id UUID -> elements(id),     -- NULL for non-relations              │
│   properties JSONB, deleted_at, created_at, updated_at                       │
│ )                                                                            │
│ -- kind: 'entity', 'relation', 'code', 'category', 'memo', 'map', 'document' │
│ -- Relations have source_id/target_id populated. Source/target can be        │
│ relations themselves.                                                        │
│                                                                              │
│ -- Context-bound properties (Sünkel/Dewey: properties are not intrinsic)     │
│ element_aspects (                                                            │
│   id UUID PK, element_id -> elements, context_id -> elements,                │
│   properties JSONB, created_at, updated_at,                                  │
│   UNIQUE(element_id, context_id)                                             │
│ )                                                                            │
│ -- An element on Map A has different properties than on Map B.               │
│ -- "Remove all relations and the entity has no content."                     │
│                                                                              │
│ Supporting Tables                                                            │
│                                                                              │
│ users (id, username, email, password_hash, display_name, role, created_at)   │
│ sessions (id, user_id, token, expires_at)                                    │
│ projects (id, name, description, created_by, created_at)                     │
│ project_members (project_id, user_id, role)                                  │
│ document_content (element_id PK, full_text, file_path, mime_type, file_size, │
│  thumbnail_path)                                                             │
│ annotations (id, project_id, code_id, document_id, constituted_by,           │
│ anchor_type, anchor JSONB, comment)                                          │
│ memo_content (element_id PK, content TEXT, format TEXT)                      │
│ memo_links (id, memo_id, target_id, link_type)                               │
│ code_hierarchy (parent_id, child_id, position)                               │
│ ai_interactions (id, project_id, event_id, request_type, model,              │
│ input_context, response, accepted, tokens_used)                              │
│                                                                              │
│ Full-text search via to_tsvector GIN indexes on document_content.full_text   │
│ (German + English).                                                          │

│ Visualization Architecture                                                   │
│                                                                              │
│ Dual-Layer Approach                                                          │
│                                                                              │
│ ┌─────────────────┬─────────────────────┬─────────────────────────────────┐  │
│ │    Map Type     │      Renderer       │               Why               │  │
│ ├─────────────────┼─────────────────────┼─────────────────────────────────┤  │
│ │ Situational Map │ DOM/SVG             │ Text-heavy, needs native text   │  │
│ │                 │ (InfiniteCanvas)    │ editing, CSS, accessibility     │  │
│ ├─────────────────┼─────────────────────┼─────────────────────────────────┤  │
│ │ Social          │ DOM/SVG             │ Elliptical regions need SVG,    │  │
│ │ Worlds/Arenas   │ (InfiniteCanvas)    │ labels need DOM                 │  │
│ ├─────────────────┼─────────────────────┼─────────────────────────────────┤  │
│ │ Positional Map  │ DOM/SVG             │ Axis rendering + scatter-plot   │  │
│ │                 │ (InfiniteCanvas)    │ positions                       │  │
│ ├─────────────────┼─────────────────────┼─────────────────────────────────┤  │
│ │ Network Map     │ Canvas API (Konva)  │ Force-directed animation needs  │  │
│ │                 │                     │ 60fps for larger graphs         │  │
│ └─────────────────┴─────────────────────┴─────────────────────────────────┘  │
│                                                                              │
│ Four Map Types                                                               │
│                                                                              │
│ 1. Situational Map — Two modes:                                              │
│ - Messy mode: Brainstorm all elements (human, nonhuman, discursive,          │
│ political/economic, temporal, spatial, symbolic, major issues). Free         │
│ placement, no connections yet.                                               │
│ - Ordered mode: Draw relations between elements. Each connection IS an       │
│ element in the data model. Annotate nature of each relation.                 │
│                                                                              │
│ 2. Social Worlds/Arenas Map — Resizable SVG ellipses for social worlds and   │
│ arenas. Overlap = shared membership. Elements placed inside worlds by        │
│ position.                                                                    │
│                                                                              │
│ 3. Positional Map — 2D axes (configurable: continuous/categorical).          │
│ Positions (stances, not persons) plotted as points. Can mark "positions NOT  │
│ taken" (dotted outline).                                                     │
│                                                                              │
│ 4. Network Map — d3-force simulation or manual layout. Nodes + edges.        │
│ Supports meta-relations.                                                     │
│                                                                              │
│ Meta-Relation Visualization (the Eta-Beziehung)                              │
│                                                                              │
│ When a relation R1 is itself related to something via R2:                    │
│ 1. R1 renders as a bezier curve between its endpoints                        │
│ 2. A diamond marker appears at R1's midpoint                                 │
│ 3. R2 renders as a curve from that diamond to its other endpoint             │
│                                                                              │
│ This keeps the visual language consistent: you always connect "nodes," even  │
│ when a "node" lives on a line.                                               │
│                                                                              │
│ Real-Time Collaboration                                                      │
│                                                                              │
│ Client A  <--WebSocket-->  WS Server  <--WebSocket-->  Client B              │
│      |                        |                            |                 │
│    Yjs Doc              Yjs Doc (auth)                  Yjs Doc              │
│    IndexedDB            PostgreSQL                      IndexedDB            │
│                                                                              │
│ - Yjs CRDTs for continuous editing (dragging elements, typing in memos)      │
│ - Discrete operations (creating codes, applying annotations) go through REST │
│  API first, persisted as events, then broadcast via WebSocket                │
│ - Room-based: map:<mapId>, document:<docId>, memo:<memoId>                   │
│ - Presence: Cursor positions, active selections, user colors                 │
│                                                                              │
│ AI Integration                                                               │
│                                                                              │
│ - All AI interactions logged in ai_interactions with full context (research  │
│ transparency)                                                                │
│ - SSE streaming for responses                                                │
│ - Every suggestion must be explicitly accepted or rejected                   │
│ - API key stored in anthropic.key (gitignored)                               │
│                                                                              │
│ Features:                                                                    │
│ 1. Coding suggestions: Selected passage + existing codes → suggested codes   │
│ with reasoning                                                               │
│ 2. Summarization: Document/section → structured summary                      │
│ 3. Pattern recognition: Annotations for a code, or map elements → patterns,  │
│ clusters, contradictions                                                     │
│ 4. Memo assistant: Draft + linked elements → analytical questions,           │
│ theoretical connections                                                      │

● This is a fundamental rethinking of the data model. Let me work through what a
   "virtual data object" would need to be.

  The current schema is still ontologically interactional: elements have a fixed
   kind, relations have fixed source_id/target_id. That distinction is baked
  into the object. But you're saying: entity vs. relation is not a property of
  the object — it's the result of perspectival collapse.

  Here's a concrete proposal:

  // The fundamental unit: a Naming (act, not substance)
  // It is NEITHER entity NOR relation intrinsically.
  // It's a superposition of both.
  interface Naming {
    id: string;
    inscription: string;    // the mark — what was uttered/inscribed
    created_at: string;
    created_by: string;
  }

  // The relational fabric: undirected, untyped bonds
  // A bond is itself a Naming (it's in the namings table too).
  // It records: "these namings participate in each other"
  // but imposes NO direction, NO roles, NO hierarchy.
  interface Participation {
    naming_id: string;      // the naming that has participants
    participant_id: string;  // the participating naming
    // That's it. No role, no direction. Pure co-constitution.
  }

  // The collapse: how a Naming appears FROM a Perspective.
  // A Perspective is itself a Naming.
  // This is where entity/relation/constellation EMERGES.
  interface Appearance {
    naming_id: string;       // the naming being observed
    perspective_id: string;  // also a naming — the observing perspective

    // The collapse mode
    mode: 'entity' | 'relation' | 'constellation' | 'process' | 'silence';

    // If mode='relation': directed READING of participations
    // (which participant appears as source, which as target)
    directed_reading?: {
      from: string;   // naming ID
      to: string;     // naming ID
      valence?: string; // "enables", "constrains", "constitutes"...
    };

    // Emergent properties (exist ONLY under this perspective)
    properties: Record<string, unknown>;
  }

  The critical differences from the current model:

  1. No kind field on the naming itself. A naming that appears as "entity" from
  one perspective can appear as "relation" from another — or as "constellation"
  when you zoom in.
  2. Participations replace source/target. The raw relational fabric is
  symmetric. Direction only appears in an Appearance. The same set of
  participations can be read differently from different perspectives.
  3. Perspectives are namings. A map is a perspective. A code is a perspective.
  An analytical framework is a perspective. Each one collapses the superposition
   differently. And each perspective can itself be observed from another
  perspective (= inquiry into the conditions of your own observation).
  4. No context. Clarke's point is honored: there's no context_id wrapping
  things. Instead, a perspective produces the appearance. The "same" naming can
  appear in indefinitely many perspectives without any of them being the primary
   "context."
  5. mode: 'silence' — a perspective can explicitly record that something is not
   appearing (positional maps: "positions not taken").

  In SQL:

  namings (
    id UUID PK,
    inscription TEXT NOT NULL,
    created_at TIMESTAMPTZ DEFAULT now(),
    created_by UUID -> users
  )

  participations (
    naming_id UUID -> namings,
    participant_id UUID -> namings,
    created_at TIMESTAMPTZ DEFAULT now(),
    PRIMARY KEY (naming_id, participant_id)
  )

  appearances (
    naming_id UUID -> namings,
    perspective_id UUID -> namings,
    mode TEXT NOT NULL,
    directed_from UUID -> namings,
    directed_to UUID -> namings,
    valence TEXT,
    properties JSONB DEFAULT '{}',
    PRIMARY KEY (naming_id, perspective_id)
  )

  Three tables. That's the entire ontology.

  What zooming in looks like: You observe naming N from perspective P. It
  appears as entity. You create a new perspective P' (also a naming) and ask:
  what are N's participations? From P', each participation might appear as a
  relation, and N dissolves into a constellation. P' is an act of inquiry into
  the arbitrariness of P.

  What a "map" is: A map is a naming. It serves as a perspective. All namings
  that have an appearance under that perspective are "on the map." Their
  positions, visual properties, modes — all in the properties JSONB of their
  appearance.

  What a "code" is: Also a naming. When you "apply a code to a passage," you
  create a participation between the code-naming and the passage-naming. From
  the coding perspective, the code appears as entity and the passage appears as
  entity, and the participation can be read as a directed relation ("codes",
  "describes", "questions").

  Should I refactor the schema to this model? It would mean rewriting
  001_initial_schema.sql and the query layer, but the API routes and UI can stay
   largely the same — they just read from appearances instead of elements.
